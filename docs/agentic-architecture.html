Stage Trigger & Dispatch

When a stage execution begins, the Blueprint backend resolves parameters and calls the Context Fabric governed run loop via REST API.

  • Trigger endpoint: POST /execute
  • Orchestrated in: orchestrator.ts
  • Payload carries: phase_model_aliases, tool_policy, auto_localize

Workbench ➔ Context Fabric Protocol

The Express API in Workgraph-Studio (under orchestrator.ts) dispatches a JSON payload outlining rules, context files, and phase configurations:

{ "stage_key": "loop.stage.code", "task": "Implement containsACharacter operator. It should check a string has a particular char or not...", "model_alias": "claude-3-5-sonnet", "phase_model_aliases": { "PLAN": "claude-3-5-sonnet", "ACT": "claude-3-5-haiku", "VERIFY": "claude-3-5-haiku", "REPAIR": "claude-3-5-sonnet" }, "tool_policy": "MUTATION", "auto_verify": true }

Context Fabric responds with a structured Stage Execution result capturing state, histories, and receipts. The details block parses failure reasons defensively:

{ "stop_reason": "FINALIZED", // or VALIDATION_BLOCKED, LLM_ERROR, etc. "error_code": "PLAN_RECEIPT_INVALID", // surfaces upstream error codes "error_message": "missing required field target_files", "final_state": { "current_phase": "FINALIZE", "repair_attempts": 1, "receipts": { "PLAN": [...], "ACT": [...] } } }

WebSocket Bridge Mode (Zero-IP Laptop Dev)

In traditional architectures, the orchestrator needs to hit a local developer’s laptop via HTTP calls (requiring static IPs, port forwarding, or tunnels).

Singularity implements a reverse WebSocket bridge inside the MCP Server setup. When a workflow execution triggers a tool run on a local workstation:

  • The laptop-side mcp-server dials outward to Context Fabric via WebSocket.
  • It establishes a persistent connection bound to the session owner's identity.
  • When Context Fabric needs to execute a tool, it routes the payload over this open channel.
  • Result: Zero firewall adjustments, zero laptop static IP setups, and 100% secure execution.
