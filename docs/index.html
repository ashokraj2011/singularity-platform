Singularity Platform

An enterprise AI-agent platform of independently-deployable services — identity, agent registry, prompt composition, LLM cost optimization, workflow orchestration, MCP execution, and a unified portal — wired together by a federated lookup layer, a cross-service audit ledger, and inline governance gates.

Overview #

The platform is a monorepo of ~19 services grouped into five planes (control, orchestration, composition, data, optimization). Every component is independently deployable; together they let you:

  • Design multi-agent workflows in a visual editor with 22+ node types
  • Govern agents through a derivation model — common locked baselines beget capability-scoped children
  • Execute agent loops against any LLM via per-tenant MCP servers
  • Observe per-step timing, cost, tokens, and artifacts on a single run insights dashboard
  • Enforce budgets and rate limits pre-flight, before any LLM call is dispatched
Where the platform earns its keep
Most agent stacks ship the happy path: prompt, model, response. This one ships the governance plane — derivation lineage, lock semantics, audit-by-default, cost gates, and a single composite endpoint that joins a run with its steps, artifacts, and ledger events. That's what makes it production-shaped, not demo-shaped.

Quickstart #

From a fresh clone to a live demo in about five minutes.

Prerequisites

  • Docker Desktop (Compose v2)
  • git, curl, psql (optional)
  • Free ports: 3000, 5174, 5175, 5180, 7100, 8000-8003, 8080, 8100-8101, 8500, 5432-5436, 9000-9001
  • ~6 GB free RAM allocated to Docker

1 · Clone

git clone https://github.com/ashokraj2011/singularity-platform.git
cd singularity-platform

2 · Boot everything

./singularity.sh up

One command brings up all three stacks: the master stack (19 services), pseudo-IAM (port 8101), and audit-governance (port 8500). First boot pulls images + builds web bundles — wait 3–5 minutes.

3 · Seed the agent-and-tools DB

( cd agent-and-tools/apps/agent-runtime \
  && DATABASE_URL="postgresql://postgres:singularity@localhost:5432/singularity" \
     npx prisma db seed )

Lands 9 agent templates (4 locked common baselines + 5 derived), 5 prompt profiles, 7 layers, 4 tools, 1 default capability.

4 · Smoke check

for u in \
  "http://localhost:8101/health" \
  "http://localhost:8500/health" \
  "http://localhost:7100/health" \
  "http://localhost:8000/health" \
  "http://localhost:8080/health" \
  "http://localhost:3000/api/runtime/agents/templates?scope=common&limit=3" \
  "http://localhost:5174/"; do
  printf "%-65s %s\n" "$u" "$(curl -s -o /dev/null -w '%{http_code}' $u)"
done

You should see 200 for all seven.

5 · Tear down

./singularity.sh down     # keep data volumes
./singularity.sh nuke     # wipe all volumes (confirmation required)

The five planes #

A useful mental model when deciding which app should own a new feature.

┌───────────────────────────────────────────────────────────┐
│ CONTROL          IAM + UserAndCapabillity + Portal        │
│  decides WHO can do WHAT; humans configure here           │
└───────────────────────────────────────────────────────────┘
                            │
┌───────────────────────────▼───────────────────────────────┐
│ ORCHESTRATION    workgraph-studio                         │
│  decides WHEN and IN WHAT ORDER work happens              │
└───────────────────────────────────────────────────────────┘
                            │
┌───────────────────────────▼───────────────────────────────┐
│ COMPOSITION      prompt-composer (inside agent-and-tools) │
│  decides WHAT TO SAY to the LLM (layered prompts +        │
│  workflow context + tool contracts + artifacts)           │
└───────────────────────────────────────────────────────────┘
                            │
┌───────────────────────────▼───────────────────────────────┐
│ DATA             agent-and-tools (registries, memory)     │
│  the WHAT of the work (templates, tools, knowledge)       │
└───────────────────────────────────────────────────────────┘
                            │
┌───────────────────────────▼───────────────────────────────┐
│ OPTIMIZATION     context-fabric                           │
│  makes LLM calls cheaper (compaction + cost tracking)     │
└───────────────────────────────────────────────────────────┘

Each plane is a layer of responsibility, not strictly a layer in a request-trace sense. A single user click can cross all five (e.g. clicking "Run" in the SPA: control plane authenticates, orchestration starts the DAG, composition assembles the prompt for the first AGENT_TASK, data plane resolves the agent template, optimization plane caches the response).

End-to-end data flow #

A typical AGENT_TASK execution traces this path:

workgraph-api (AgentTaskExecutor)
    │
    │ POST /execute  { task, run_context, capability_id, agent_template_id }
    ▼
context-fabric (context-api on :8000)
    │
    │ 1. POST prompt-composer/compose-and-respond  (previewOnly=true)
    │    → returns assembled prompt + tool descriptors
    │
    │ 2. context-memory/memory/search → relevant prior facts
    │
    │ 3. POST mcp-server/mcp/invoke
    │      ↓ pre-flight: GET audit-gov/governance/budgets/check
    │      ↓ pre-flight: GET audit-gov/governance/rate-limits/check
    │      ↓ LOOP { llm call → tool call (via tool-service or local) }
    │      ↓ emit-as-you-go: llm.call.completed, tool.execution.*,
    │                        governance.denied, approval.wait.created
    │
    │ 4. POST audit-gov/events  (cf.execute.completed with traceId)
    │
    ▼
workgraph-api persists results → next node fires

Every step writes to the central audit_events table keyed by trace_id (which equals the workflow_instance_id for workgraph-driven runs). The run insights endpoint joins these events back into the per-step view.

Agent Studio (M23) #

A single workbench surface for governance-aware agent authoring. Replaces the flat /agents and /agent-templates admin pages.

The derivation model

Every AgentTemplate falls into one of two states:

StateIdentifierEditability
Common / Locked capabilityId IS NULL, lockedReason set Platform-admin only
Derived capabilityId set, baseTemplateId points at a parent Owner of the capability (or platform-admin)
Custom capabilityId set, baseTemplateId IS NULL Owner of the capability (or platform-admin)

The four seeded baselines — ARCHITECT, DEVELOPER, QA, GOVERNANCE (UUIDs 00000000-0000-0000-0000-0000000000d1..d4) — are common locked templates. Each ships its own basePromptProfileId wired to a layered prompt (platform constitution + role contract + output contract).

Schema delta (M23)

model AgentTemplate {
  // ... existing fields ...

  // M23 — Agent Studio governance fields
  capabilityId    String?
  baseTemplateId  String?
  lockedReason    String?

  baseTemplate     AgentTemplate?  @relation("AgentTemplateDerivation", fields: [baseTemplateId], references: [id])
  derivedTemplates AgentTemplate[] @relation("AgentTemplateDerivation")

  @@index([capabilityId])
  @@index([baseTemplateId])
}

Authorization model

Three helpers anchor authorization on the actor's JWT claims (capability_ids[], roles[], is_platform_admin, is_super_admin):

  • isPlatformAdmin(actor) — checks role/flag
  • canManageCapability(actor, capabilityId) — checks capability_ids membership or capability-owner:<id> role
  • requireCapabilityOwner(actor, capabilityId, action) — throws ForbiddenError if neither

Mutation routes are gated by requireAuth middleware; read routes stay anonymous (the controller decorates each template with editable: false when there's no user). Cross-capability derivation (deriving from a base that belongs to a different capability) is blocked unless the actor is platform-admin.

UI surface

  • Capability picker (dropdown of capabilities from /api/runtime/capabilities) — UUID-validated
  • Two grouped sections: Capability Agents (editable, lineage badge), Common Library (locked badge)
  • Right detail panel: lineage, scope, lock status, base prompt profile id with rendered prompt-layer tree (priority-sorted, layerType/scopeType badges)
  • Derive dialog: name + description override; creates a capability-scoped child inheriting role + prompt profile + tool policy

Run Insights (M24) #

A single composite endpoint that joins a workflow run with its per-step timing, artifacts, and cost. Visualised as a Gantt-style timeline at /runs/:id/insights.

The composite endpoint

GET /api/workflow-instances/:id/insights

Returns:

  • run: id, name, status, startedAt, completedAt, durationMs
  • totals: nodes count + breakdown by status, documents, consumables, llm_calls, total_tokens, total_cost_usd, governance_denied
  • nodes[]: per-step status, startedAt/completedAt, durationMs, durationPrecise (true when sourced from the M24.5 columns), linked documents, linked consumables, event count
  • documents[], consumables[]: full artifact list with MinIO refs
  • costByModel[]: provider/model breakdown sourced from audit-gov
  • events[]: full audit timeline (subject_id + trace_id joins to capture cross-service events)

Per-step timing precision (M24.5)

The original M24 inferred per-step duration as updatedAt - createdAt on completed nodes. M24.5 added two columns to WorkflowNode:

// workgraph-studio/apps/api/prisma/schema.prisma
model WorkflowNode {
  // ...
  startedAt    DateTime?  // stamped on PENDING → ACTIVE
  completedAt  DateTime?  // stamped on COMPLETED / FAILED / SKIPPED
}

The central WorkflowRuntime plus six executors (DecisionGate, ErrorCatch, Foreach, ParallelFork, ParallelJoin, PolicyCheck) and the approvals router write these columns on every status transition. The insights handler prefers them, falling back to the old heuristic only for runs that pre-date the schema change. The SPA renders precise durations without the ≈ approximate marker.

UI surface

  • 7 summary tiles: total duration, steps (with status counts), LLM calls, tokens, total cost USD, documents, consumables, governance denials
  • Per-step Gantt: horizontal bars sized by duration, status-coloured, with badges showing 📎 documents / 📦 consumables / ⚡ events per node
  • Artifacts table: documents + consumables with downloadable MinIO refs
  • Cost-by-model breakdown: provider / model / calls / tokens / $ per row
  • Audit timeline: severity-coloured event log keyed to the run
  • 5s polling so the dashboard tracks in-progress runs live

Entry: a green Insights → pill in the RunViewerPage header.

Pre-flight enforcement

Before each LLM dispatch, the mcp-server agent loop awaits two parallel checks:

// mcp-server/src/lib/audit-gov-check.ts
const [budgetRes, rateRes] = await Promise.all([
  checkBudget(state.correlation.capabilityId, undefined),
  checkRateLimit(state.correlation.capabilityId, undefined),
])
if (!budgetRes.allowed) {
  emitAuditEvent({ kind: "governance.denied", ... })
  return { kind: "denied", finishReason: "governance_denied", reason, check: "budget", ... }
}

The response carries status: "DENIED" with the failing check + remaining-window detail. The check is fail-open — an unreachable audit-gov defaults to allow so a brownout doesn't stall every tenant — but the failure is logged at warn level.

SPA surfaces

  • localhost:3000/audit — recent events, severity-coloured, filterable by source_service / kind
  • localhost:3000/cost — cost rollup by capability, period (day/week/month), with model breakdown

Execution location

Each node has an executionLocation that determines where its executor runs:

  • SERVER — Express runtime fires the executor directly
  • CLIENT — browser engine via BrowserWorkflowRuntime
  • EDGE / EXTERNAL — claim-token pattern for async external runners

HUMAN_TASK file uploads

Uploads from browser-driven /play sessions write to MinIO. The Document model has FKs onto WorkflowNode and WorkflowInstance, but browser-play uses design-time node IDs and IndexedDB-local run IDs that don't exist as those rows. The handler at POST /api/documents/upload looks up the supplied IDs first and nulls them when missing, so uploads succeed regardless of the run-source.

Widget-less HUMAN_TASK nodes (the common "review attachments and mark done" pattern) still render a "Submit & complete" button even when the form has zero fields — so the assignee can always advance the workflow.

Troubleshooting #

Boot

"docker compose v2 not found"
Docker Desktop isn't running or only legacy docker-compose v1 is installed. docker version verifies the daemon is up; docker compose version verifies the v2 plugin. On managed Macs you may need to install Docker Desktop via corporate self-service.
Two Postgres servers on :5432
Stop the host's Postgres or remap at-postgres's host port — the master compose binds 5432 to the agent-and-tools DB by default.
Workgraph rejects JWT (401s on every endpoint)
JWT secret divergence — workgraph-api enforces a 32+ char secret. The compose defaults align pseudo-iam, workgraph-api, and IAM at dev-secret-change-in-prod-min-32-chars!!. Verify with docker exec docker-api-1 printenv JWT_SECRET and the same on pseudo-iam.

Auth

/context-picker → /dashboard → /login loop
Token verifies but downstream IAM rejects it. Ensure workgraph-api's IAM_BASE_URL points at pseudo-IAM (:8101), not real IAM (:8100) — the lookup proxy forwards the user JWT and the upstream IAM must be the same authority.
/api/runtime/inbox or /api/agent-studio 500
Workgraph-api's local User mirror has no row for the pseudo-IAM sub. AUTH_PROVIDER=iam triggers the lazy mirrorIamUser at first sight; if you cold-path it via a tool call, the mirror won't have run yet. Hard-refresh the SPA to mint a fresh JWT and have the auth middleware do its thing.

Seed / DB

Prisma seed fails with "no unique or exclusion constraint"
Prisma's compound-key upsert codegen occasionally emits an ON CONFLICT shape Postgres rejects after a db push --force-reset. The seed uses an explicit findFirst + create/update pattern to sidestep that. If you hit it after future schema changes, replace any upsert({ where: { compound_key: ... } }) with the explicit pattern.
"type vector does not exist" after force-reset
pgvector extension was wiped. Recreate it: 
docker exec agentandtools-postgres psql -U postgres -d singularity -c \
  "CREATE EXTENSION IF NOT EXISTS vector;"

Files

File upload from /play returns 500 (FK violation)
Browser-driven runs use design-time node IDs and IndexedDB-local run IDs. The upload handler nulls supplied IDs that don't exist in WorkflowNode / WorkflowInstance, so this should "just work" in current code. If it doesn't, check Document table FKs are ON DELETE SET NULL.
HUMAN_TASK has no Complete button
Widget-less form bodies render the "Submit & complete" button as a primary action. If a node has zero widgets defined at design-time, this is the expected path. Make sure the user has claimed the node first.
