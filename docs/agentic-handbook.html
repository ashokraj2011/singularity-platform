Agentic Programming & Planning Handbook

A detailed architectural look at how agentic programming is coordinated on Singularity, using context-fabric as the stateful, governed brain.

Why Context-Fabric is the Real Player

In many agentic frameworks, the developer loop is hosted in the same process that executes file updates and terminal commands. In the Singularity Platform, we enforce a strict Security Segregation Rule: "Business engines orchestrate; Sandboxes execute."

The mcp-server (the sandbox) is a completely stateless executor. It has no understanding of context, the current stage of the ticket, the conversation history, or the safety bounds of the files it modifies. It simply accepts targeted, sandboxed instructions (e.g., "replace this block of text" or "run this test script") and reports back with results.

The **Real Player** is the context-fabric microservice. Built as a Python FastAPI suite under context-fabric/services/context_api_service, it owns all state, history, planning rules, masking logic, and quality checks.

State & History
Keeps the multi-turn memory buffer alive across restarts, compaction iterations, and manual approval pauses using the call_log and events_store databases.
👮
Phase Governance
Translates free-form LLM outputs into structured, validated phase receipts and denies execution of files that bypass the active policy.

Maintains a secure, stateful key-value token mapping across turns so the LLM never communicates raw emails or API keys with providers.

The Architectural Shift: Legacy invite.ts vs. context-fabric

Prior to Milestone 71, the entire LLM ↔ tool execution loop was housed in a large ~4,600-line monolithic TypeScript endpoint at mcp-server/src/mcp/invoke.ts. In modern Singularity, the cloud POST /mcp/invoke route returns 410 Gone (re-enableable only via MCP_LEGACY_INVOKE_ENABLED for incident recovery); the loop body is retained but dormant for the platform path. The active loop was refactored into the Python-based context-fabric/services/context_api_service/app/governed modules.

Deep Dive: How Agentic Coding is done in the Context Planner

The "Context Planner" is the governance framework that maps the progression of the agent from an empty slate to a verified code change. An agent cannot write code or inspect files arbitrarily without declaring its plan in a structured PlanReceipt.

The core planning lifecycle goes through the following structured phases, mapped in phase_state.py:

The Structural Phase Map
PLAN ➔ EXPLORE ➔ ACT ➔ VERIFY ➔ [REPAIR] ➔ SELF_REVIEW ➔ FINALIZE

1. The Planning Stage (PLAN Phase & PlanReceipt)

When a stage begins, the agent is placed in the PLAN phase. Its tool allowlist is highly restricted (typically limited to read_file or repo_map). The agent must submit a PlanReceipt.

Key fields in the Pydantic PlanReceipt structure (defined in receipts.py):

  • target_files: The list of files the agent expects to edit. This field triggers subsequent path-coverage validations.
  • symbols_to_inspect: Symbols/classes to analyze.
  • test_strategy: Contains commands, which is the list of scripts/tests the agent intends to run to verify its work. Mandatory for code stages.
  • actionable: Can be "yes", "no" (story premise already satisfied), or "blocked". Short-circuits the run to human verification if "no" or "blocked", avoiding fabricated tool execution.

2. Role-Specific Overrides

To prevent validation failures when running non-coding tasks (e.g. security audits or release-readiness reviews), receipts.py overrides planning schemas based on the agent_role:

  • Reviewer Roles (SECURITY, DEVOPS): Uses ReviewPlanReceipt. The test_strategy becomes optional (since auditors do not run tests), and a review_strategy is introduced.
  • Intake Roles (PRODUCT_OWNER): Uses StoryIntakeReceipt. The fields shift entirely to story_brief (markdown) and acceptance_criteria (pass/fail assertions).

3. Enforcement & Code Change Provenance

Once a plan is approved and the agent transitions through EXPLORE (reads context) to ACT (applies changes), two strict validation filters run in the background inside loop.py:

A. Path Coverage Gap (check_path_coverage in path_coverage.py)

Asserts that every file listed in the plan's target_files is covered by either an edit block in the EditReceipt.edits array or listed in the EditReceipt.skipped_targets list. Unplanned files can be touched (over-coverage is allowed), but missing planned targets triggers a PHASE_COVERAGE_GAP error.

B. Edit Provenance (PHASE_EDIT_UNBACKED)

To prevent the agent from submitting a fake receipt claiming it edited files without executing mutating tools, loop.py tracks every successful tool invocation of apply_patch, replace_text, etc., and extracts their code_change_id or tool_invocation_id. If an entry in the EditReceipt is not backed by one of these recorded tokens, it is rejected as PHASE_EDIT_UNBACKED.

Submitted during the ACT phase. Validates file modifications. Triggers path-coverage & edit-provenance check.

VerificationReceipt Schema & Guards

Submitted during the VERIFY phase. Confirms test outputs. Prevents the "fake-pass" and "failing-test-bypass" loopholes.

M74 Quality Bar Restorations

Milestone 74 restored several vital verification gates that were disabled during structural refactoring. These prevent models from bypassing safety policies:

History Compression (`history_compression.py`)

To keep LLM context consumption stable, older conversation turns are summarized as single user-role messages ([TURN-N-RECAP]) while preserving the last 8 turns verbatim. 

def _count_breadcrumbs(prelude: list[dict[str, Any]]) -> int:
    count = 0
    for msg in prelude:
        if msg.get("role") != "user":
            continue
        content = msg.get("content") or ""
        if content.startswith("[TURN-") and "-RECAP]" in content:
            count += 1
    return count

Edit Provenance Binding (`loop.py`)

Ensures every path claimed in EditReceipt matches a registered, successful mutation call. 

def _check_edit_receipt_provenance(*, receipt: dict, produced_changes: dict) -> list[str]:
    edits = receipt.get("edits") or []
    unbacked = []
    for entry in edits:
        path = entry.get("file")
        canonical = _normalise_change_path(path)
        if canonical not in produced_changes or not produced_changes[canonical]:
            unbacked.append(path)
    return unbacked

Auto-Verify & Log Preservation (`verify_synthesis.py`)

Automatically fires test runs on behalf of the agent. To prevent truncation from discarding actual test failures, the framework preserves the **tail** (stderr and stack traces) rather than the head. 

def _summarise(text: Any, *, max_len: int = 1500) -> str:
    s = str(text)
    if len(s) <= max_len:
        return s
    # Drop standard setup logs, keep actual error traces at the end
    return "...[truncated earlier logs]\n" + s[-max_len:]

Validation Retry Budget (`stage_driver.py`)

Validation errors (like a missing field or incorrect receipt format) no longer abort the stage run instantly. The agent receives a structured correction prompt and has a budget of 2 consecutive retries to self-correct. 

if turn.step.validation_error and not turn.step.phase_advanced:
    consecutive_validation_errors += 1
    if consecutive_validation_errors > _VALIDATION_RETRY_BUDGET:
        result.stop_reason = "VALIDATION_BLOCKED"
        return result
    # Inject validation errors directly as a user message
    history.append(_render_validation_error_message(turn.step.validation_error))
    continue

The PII Masking Engine (pii_mask.py)

Because LLM provider dispatches are external, context-fabric runs a stateful de-identification pass around all tool inputs and outputs.

Args Unmasking

Recursively inspects the tool call arguments and replaces tokens (e.g. [EMAIL_1]) with their raw values. Sorting by length in reverse prevents substring mismatch corruption (e.g. replacing [EMAIL_1] inside [EMAIL_10]).

Result Masking

Scans incoming stdout/results using regular expressions for IPv4, emails, phone numbers, credit card digits (validated by Luhn check), and SSNs, converting them to stable sequential tokens.
